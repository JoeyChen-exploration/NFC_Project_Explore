# LinkHub 下一阶段优化路线图（v1）

> 目标：先保证“稳定可用”，再做“效率与增长”。
> 适用时间：当前 PR 合并后到下一轮里程碑发布。

## 优先级总览

1. Draft / Published 发布态开关（最高优先级）
2. 一键公网写卡流程（Cloudflare Tunnel 集成）
3. 前端实时校验与可用性反馈
4. 公开页性能与 SEO 基础
5. 自动化测试最小闭环

---

## 1) Draft / Published 发布态开关（P0）

### 目标

- 避免编辑中的改动被实时暴露到公开页。
- 让用户有明确“发布”动作，降低误操作风险。

### 范围

- 数据层：区分 `draft` 与 `published` 两套数据（至少 profile/socials/links）。
- 接口层：新增发布接口（例如 `POST /api/profile/publish`）。
- 展示层：公开页只读取 `published` 数据。
- 编辑器：新增状态标识（草稿未发布 / 已发布）与“发布更改”按钮。

### 验收标准

- 在编辑器修改资料后，不点击发布，公开页保持不变。
- 点击发布后，公开页一次性同步为新内容。
- 发布失败时有清晰错误提示，不覆盖旧发布内容。

### 风险与注意事项

- 迁移旧用户数据时要有默认发布快照，避免公开页空白。
- 需要处理并发发布（短时间重复点击）。

---

## 2) 一键公网写卡流程（P1）

### 目标

- 减少从本地开发到手机实测的手工步骤。
- 避免把 `localhost` 写进 NFC 卡。

### 范围

- Dashboard 增加“生成公网地址”入口。
- 自动拼接 `/nfc/:serial` 并提供一键复制。
- 增加写卡前校验：检测链接是否为公网域名。

### 验收标准

- 用户 3 步内完成“生成公网地址 -> 复制写卡链接 -> 手机可访问”。
- 若链接是 `localhost`，界面阻止写卡并提示修复方式。

---

## 3) 前端实时校验与可用性反馈（P1）

### 目标

- 把错误前置到输入阶段，减少“点保存才报错”的挫败感。

### 范围

- 社交字段实时规范化提示（`username` / `@username` / URL）。
- 链接字段格式与可访问性基础校验。
- 统一错误信息文案（中英文一致）。

### 验收标准

- 用户在输入框阶段即可知道格式是否可提交。
- 保存报错率明显下降（以日志或主观反馈衡量）。

---

## 4) 公开页性能与 SEO 基础（P2）

### 目标

- 提升扫码后首屏速度与分享可读性。

### 范围

- 补充 `title / description / og:*` 基础 meta。
- 公开页关键资源优化（图片/脚本加载策略）。
- 页面骨架与空状态优化。

### 验收标准

- 公开页首屏渲染更稳定（低网速下可感知改善）。
- 分享到社交软件时，预览信息完整可读。

---

## 5) 自动化测试最小闭环（P1）

### 目标

- 保障核心链路可回归，避免迭代后反复回退问题。

### 范围（最小集）

- 社交输入规范化单测（`@name` / `name` / URL）。
- 编辑保存后与公开页一致性测试（核心 API + 页面）。
- NFC 跳转链路测试（`/nfc/:serial`）。

### 验收标准

- CI 能跑通最小测试集。
- 核心功能改动时至少覆盖一条对应测试。

---

## 执行策略

- 先做第 1 条（发布态开关），再并行推进第 3 与第 5 条。
- 第 2 条依赖开发环境与产品交互设计，可在第 1 条后开始。
- 第 4 条放在核心链路稳定后补齐。
